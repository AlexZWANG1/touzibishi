# Iris 改造方案反馈 — 一次性执行版

> 基于对 iris/ 全部源码的审查，以下是对改造方案的修正意见。请按此文档执行，不分期。

---

## 一、方案中正确的部分（直接执行）

### 1. 硬预算治理（budget.py）
方案正确。当前 `_memory_flush` 和 `_compact_context` 会触发额外 LLM 调用但不计入 `max_tool_rounds` / `max_total_tool_calls`，导致实际开销超预算。

**执行要求：**
- 新增 `core/budget.py`，统一计量所有 LLM 调用（主循环 + flush + summary + compaction）
- 工具执行前做"可执行额度裁剪"：如果剩余额度不够完成当前轮所有 tool call，裁剪到可执行数量
- `HarnessResult` 增加 `budget_breakdown` 字段，分类统计主循环/flush/compaction 各消耗多少

### 2. Loop Detector（loop_detector.py）
方案正确。当前只有 stagnation detection（harness.py 215-231 行，同签名重复 3 次），缺少 pingPong 和 noProgressPoll。

**执行要求：**
- 新增 `core/loop_detector.py`，实现三个检测器：
  - `generic_repeat`：同工具同参数连续重复（现有逻辑迁移过来）
  - `ping_pong`：A→B→A→B 交替模式检测
  - `no_progress_poll`：输出内容无实质变化（比较连续 N 轮工具返回的 hash 或关键字段）
- 阈值写入 `iris_config.yaml` 的 `loop_detection` 节：
  ```yaml
  loop_detection:
    generic_repeat_threshold: 3
    ping_pong_threshold: 3
    no_progress_threshold: 3
    action: "steer_then_stop"  # steer_then_stop | hard_stop | warn_only
  ```
- 检测到后先注入 steering message（"你在重复操作，请换一个思路"），如果下一轮仍然触发则 hard stop
- 从 harness.py 删除现有的 `_last_tool_sig` / stagnation 逻辑，统一到 loop_detector

### 3. 按主题注入历史
方案正确。当前 `_load_prior_context`（harness.py 510-545 行）全量注入最近 3 假设 + 10 观察，没按当前 query 的 subject 过滤。

**执行要求：**
- 改 `_load_prior_context`：先从 user_input 提取 subject（公司名/ticker），然后只注入该 subject 相关的 hypotheses 和 observations
- 不相关的历史降级为一句话摘要（"此前还研究过 XX、YY，如需可用 query_knowledge 查询"）
- 改 `retrieval.py` 的查询方法，增加 `by_subject(subject: str)` 过滤接口

### 4. Tool Hook 机制（tool_hooks.py）
方案正确。当前工具异常在 `_dispatch_tool_calls` 里被 catch 后直接返回 error string，没有统一的审计/降级/阻断接口。

**执行要求：**
- 新增 `core/tool_hooks.py`：
  ```python
  class ToolHookContext:
      tool_name: str
      args: dict
      run_id: str
      round_number: int
      budget_remaining: dict

  class ToolHooks:
      def before_tool_call(self, ctx: ToolHookContext) -> ToolHookContext | None:
          """返回 None 则阻断。可修改 args 做参数规范化。"""
      def after_tool_call(self, ctx: ToolHookContext, result: ToolResult) -> ToolResult:
          """可修正结果、打审计标签、触发降级。"""
  ```
- 把 `_dispatch_tool_calls` 的 try/except 改为走 hook：异常时 after_tool_call 记录事件 + 返回降级结果
- 默认 hook 实现：before 做参数类型检查，after 做错误分类标签（network_error / data_error / timeout 等）

### 5. Context Pipeline 提取
方案方向正确，但**不要做成双阶段管线框架**，过度抽象。

**执行要求：**
- 从 harness.py 提取 `core/context.py`，包含一个 `ContextAssembler` 类：
  ```python
  class ContextAssembler:
      def load_prior_context(self, subject: str, retriever) -> list[dict]
      def build_system_message(self, soul: str, tools_exposed: list) -> dict
      def should_compact(self, messages: list, limit: int) -> bool
      def compact(self, messages: list, llm_client, budget) -> list[dict]
      def memory_flush(self, messages: list, knowledge_tools, budget) -> None
  ```
- harness.py 的 `_build_user_message`、`_load_prior_context`、`_compact_context`、`_memory_flush` 迁移到这个类
- harness.py 主循环调用 `self.context.xxx()` 即可，不需要管线注册机制

### 6. 事件持久化
方案说建三张表（run_jobs、run_events、session_locks），**砍到一张表**。

**执行要求：**
- 在现有 `iris.db` 中新增一张 `run_log` 表：
  ```sql
  CREATE TABLE IF NOT EXISTS run_log (
      id INTEGER PRIMARY KEY AUTOINCREMENT,
      run_id TEXT NOT NULL,
      event_type TEXT NOT NULL,
      data_json TEXT,
      created_at REAL NOT NULL DEFAULT (julianday('now'))
  );
  CREATE INDEX idx_run_log_run_id ON run_log(run_id);
  ```
- harness.py 的 `_emit` 方法同时写入这张表
- 不做 run_jobs 和 session_locks，不需要

### 7. 配置清理
**执行要求：**
- 删除 `iris_config.yaml` 中与 phase 相关的任何残留
- 新增 `loop_detection` 节（见上）
- 新增 `budget` 节：
  ```yaml
  budget:
    include_flush_in_tool_rounds: true
    include_compaction_in_tool_rounds: true
    pre_round_trim: true  # 执行前裁剪到可执行额度
  ```
- `vector_search` 节保留但 `enabled` 改为 `false`（见下面"方案遗漏"）

### 8. CLI/UI 观测增强
**执行要求：**
- main.py CLI handler 增加显示：run_id、当前轮次/总轮次预算、已用 token、loop detector 状态
- app.py Streamlit 侧边栏增加：预算进度条、工具调用计数、停止原因

## 二、方案中需要砍掉的部分

### ~~Session Queue + run_store.py~~
**砍掉。** Iris 当前是单用户 CLI/Streamlit 应用，没有多租户并发场景。session_locks、per-group 队列、串行排队机制解决的是你还没遇到的问题。当前 harness.py 的 `_abort` + `steer()` 已经够用。

如果未来有多用户需求，再加不迟。现在做是过度工程。

### ~~runtime.py 独立模块~~
**砍掉。** 理由同上。harness.py 本身就是 runtime，不需要再套一层。

### ~~容器隔离执行~~
**砍掉。** 投研工具不需要沙箱隔离。

## 三、方案遗漏 — 必须补做的

### 遗漏 1：vector_search 是空壳，必须接通或关掉

**现状：** `iris_config.yaml` 写了 `vector_search.enabled: true`、`model: text-embedding-3-small`，`retrieval.py` 有 `embeddings` 表，但 `memory_search` 工具的向量检索路径**没有实际接通**——没有调用 embedding API，没有写入 embedding 向量，检索走的是纯文本匹配。

**要求：** 二选一：
- **选项 A（推荐）：** 接通它。在 `retrieval.py` 的写入方法里调用 OpenAI embedding API 生成向量并存入 `embeddings` 表，`memory_search` 用余弦相似度检索。记得把 embedding 调用也计入 budget。
- **选项 B：** 关掉它。`vector_search.enabled` 改为 `false`，删除 `embeddings` 表相关死代码，`memory_search` 文档说明当前是关键词匹配。

不能维持现状（配置说开了，代码没接通）。

### 遗漏 2：工具结果截断阈值太激进

**现状：** `compress_threshold_chars: 2000`，所有工具返回超 2000 字符的结果都会被截断。财务数据（财报表格、多年数据对比）经常超 2000 字符，截断后丢失关键数字。

**要求：**
- 默认阈值提高到 `5000`
- 按工具类型差异化：财务工具（`fmp_financials`、`fred_series`）阈值 `8000`，搜索工具保持 `5000`
- 在 `iris_config.yaml` 增加：
  ```yaml
  compress_threshold_chars: 5000
  tool_compress_overrides:
    fmp_financials: 8000
    fred_series: 8000
  ```

### 遗漏 3：动态工具注入的关键词匹配太粗糙

**现状：** `_select_tools_for_round`（harness.py 547-617 行）用 context 子串匹配工具 schema 里的关键词来决定暴露哪些工具。容易误注入（无关工具被选中）和漏注入（该用的工具没暴露）。

**要求：**
- 短期修复：把匹配从"context 全文包含关键词"改为"最近 3 条 message 包含关键词"，减少噪音
- 给每个工具增加 `trigger_keywords` 显式配置（而不是从 schema description 里自动提取）
- 在 `iris_config.yaml` 增加：
  ```yaml
  tool_triggers:
    exa_search: ["搜索", "查找", "search", "find", "新闻", "news"]
    fmp_financials: ["财报", "收入", "利润", "revenue", "earnings", "P/E", "估值"]
    fred_series: ["宏观", "利率", "GDP", "CPI", "失业率", "macro"]
    # ...其他工具
  ```

### 遗漏 4：harness.py run() 的 initial_phase 死代码清理

**要求：**
- 删除 `run()` 签名中的 `initial_phase` 参数
- 删除 `EventType.PHASE_CHANGE` 枚举值
- 更新所有调用方（tests、main.py、app.py）移除 `initial_phase` 传参
- 这是个小改动但必须做干净，否则后续维护者会困惑

## 六、验收标准

改造完成后应满足：

1. **预算可控**：任何 run 的总 LLM 调用次数（含 flush/compaction）不超过配置上限
2. **无静默死循环**：pingPong、noProgress 场景能被检测并中断
3. **历史注入精准**：查 AAPL 时不注入 TSLA 的历史假设
4. **工具异常可审计**：每个工具调用的成功/失败/降级都记录在 run_log
5. **配置-代码一致**：vector_search 要么真的工作，要么配置关闭
6. **财务数据不丢失**：FMP 返回的完整财报数据不被截断掉关键数字
7. **phase 完全消失**：代码中搜索 "phase" 无任何残留
